Servlet简介

Servlet是sun公司提供的一门用于开发动态web资源的技术。

狭义的Servlet是指Java语言实现的一个接口，广义的Servlet是指任何实现了这个Servlet接口的类，一般情况下，人们将Servlet理解为后者。Servlet运行于支持Java的应用服务器中。从原理上讲，Servlet可以响应任何类型的请求，但绝大多数情况下Servlet只用来扩展基于HTTP协议的Web服务器。

Sun公司在其API中提供了一个Servlet接口，用户若想开发一个动态web资源(即开发一个Java程序向浏览器输出数据)，只需要完成以下2个步骤：

  1. 编写一个Java类，实现Servlet接口。
  2. 把开发好的Java类部署到web服务器中。

Servlet是如何被外界访问到的

  1. 由于客户端是通过URL地址访问web服务器中的资源，所以Servlet程序若想被外界访问，必须把servlet程序映射到一个URL地址上，这个工作在web.xml文件中使用<servlet>元素和<servlet-mapping>元素完成。
  2. <servlet>元素用于注册Servlet，它包含有两个主要的子元素：<servlet-name>和<servlet-class>，分别用于设置Servlet的注册名称和Servlet的完整类名。
  3. 一个<servlet-mapping>元素用于映射一个已注册的Servlet的一个对外访问路径，它包含有两个子元素：<servlet-name>和<url-pattern>，分别用于指定Servlet的注册名称和Servlet的对外访问路径。

Servlet的执行过程

在上面的例子中，Servlet的执行过程如下：

  1. 在浏览器上输入http://localhost/test/servlet。
  2. 被服务器这边监听80端口号的connector HTTP 1.1接收，接收之后又将请求转发给Engine处理。
  3. Engine将请求匹配给虚拟主机localhost，如果没有找到，也是匹配给它，因为它是默认的虚拟主机。
  4. 虚拟主机接收到请求，然后将请求的路径变成/test/servlet，之后在该host节点下去搜寻path=/test的Context节点。
  5. 找到test应用，将请求转发给该应用，之后再去test应用下的WEB-INF目录下的web.xml中去搜索子元素<url-pattern>的值为/test的<servlet-mapping>元素。
  6. 读取元素<servlet-mapping>的子元素<servlet-name>的值，由此确定Servlet的名字为“servlet”。
  7. 找到子元素<servlet-name>的值为servlet的<servlet>元素。
  8. 读取元素<servlet>的子元素<servlet-class>的值，由此确定Servlet的类名为ServletTest。
  9. 到Tomcat安装目录/webapps/test/WEB-INF/classes/目录下查找到ServletTest.class文件。
  10. web服务器首先检查是否已经装载并创建了该Servlet的实例对象。如果是，则直接执行第13步，否则，执行第11步。
  11. 装载并创建该Servlet的一个实例对象。
  12. 调用Servlet实例对象的init()方法。
  13. Tomcat创建一个用于封装HTTP请求消息的HttpServletRequest对象和一个代表HTTP响应消息的HttpServletResponse对象，然后调用Servlet的service()方法并将请求和响应对象作为参数传递进去。
  14. web应用程序被停止或重新启动之前，Servlet引擎将卸载Servlet，并在卸载之前调用Servlet的destroy()方法。

Servlet的生命周期

Servlet是一个供其他Java程序（Servlet引擎，例如：Tomcat）调用的Java类，它不能独立运行，它的运行完全由Servlet引擎来控制和调度。

针对客户端的多次Servlet请求，通常情况下，服务器只会创建一个Servlet实例对象，也就是说Servlet实例对象一旦创建，它就会驻留在内存中，为后续的其它请求服务，直至web容器退出(或应用停止)，servlet实例对象才会销毁。

在Servlet的整个生命周期内，Servlet的init()方法只被调用一次。而对一个Servlet的每次访问请求都导致Servlet引擎调用一次Servlet的service()方法。对于每次访问请求，Servlet引擎都会创建一个新的HttpServletRequest请求对象和一个新的HttpServletResponse响应对象，然后将这两个对象作为参数传递给它调用的Servlet的service()方法，service()方法再根据请求方式分别调用doGet()等方法。

从提高Servlet容器运行性能的角度出发，Servlet规范为Servlet规定了不同的初始化情形。如果有些Servlet专门负责在web应用启动阶段为web应用完成一些初始化操作，则可以让它们在web应用启动时就被初始化。对于大多数Servlet，只需当客户端首次请求访问时才被初始化。假设所有的Servlet都在web应用启动时被初始化，那么会大大增加Servlet容器启动web应用的负担，而且Servlet容器有可能会加载一些永远不会被客户访问的Servlet，白白浪费容器的资源。

可以在web应用的WEB-INF目录下的web.xml配置文件的<servlet>元素中配置了一个<load-on-startup>元素，那么web应用程序在启动时，就会装载并创建Servlet的实例对象、以及调用Servlet实例对象的init()方法。用途举例如下：可以为web应用写一个InitServlet，将这个Servlet配置为启动时装载，为整个web应用创建必要的数据库表和数据。

URL冲突匹配的问题

URL映射会遇到冲突匹配的问题，下面给出几个例子：

已知映射关系如下：

  1. Servlet映射到/abc/*。
  2. Servlet映射到/*。
  3. Servlet映射到/abc。
  4. Servlet映射到*.do。

冲突匹配问题如下：

  1. URL为/abc/a.html时，/abc/*和/*都匹配，此时Servlet引擎会调用Servlet1。
  2. URL为/abc时，/abc和/*都匹配，此时Servlet引擎会调用Servlet3。
  3. URL为/abc/a.do时，/abc/*和*.do都匹配，此时Servlet引擎会调用Servlet1。
  4. URL为/a.do时，/*和*.do都匹配，此时Servlet引擎会调用Servlet2。
  5. URL为/xxx/yyy/a.do时，/*和*.do都匹配，此时Servlet引擎会调用Servlet2。

URL匹配的规则

  1. “/”开头的url-pattern优先级高于“*.后缀”的优先级。
  2. 如果都是/开头的url-pattern，会按照精准匹配的原则，哪个匹配程度高，就执行哪个。
  3. /*的优先级要高于/，如果想要/起作用，则不能有/*存在。
  4. 当有/存在的情况下，html页面、图片仍然访问不到，jsp可以访问到。
  5. 访问静态资源的时候，也有servlet参与。当访问静态资源的时候，会被/匹配，而/被DefaultServlet所映射。DefaultServlet的处理逻辑就是去项目中搜索你要加载的资源，如果找到，则以流的形式写回浏览器端；如果找不到，则报404。

DefaultServlet

凡是在web.xml文件中找不到匹配的<servlet-mapping>元素的URL，它们的访问请求都将交给缺省Servlet（即DefaultServlet）处理，也就是说，缺省Servlet用于处理所有其他Servlet都不处理的访问请求。

默认的DefaultServlet

在tomcat的安装目录下的\conf\web.xml文件中，注册了一个名称为org.apache.catalina.servlets.DefaultServlet的Servlet，并将这个Servlet设置为了缺省Servlet。

当访问Tomcat服务器中的某个HTML文件、图片等静态资源时，实际上是在访问这个缺省Servlet。这个Servlet的处理逻辑就是去项目中搜索你要加载的资源，如果找到，则以流的形式写回浏览器端；如果找不到，则报404。

用户自定义DefaultServlet

用户还可以自定义一个DefaultServlet来取代tomcat自带的DefaultServlet。

如果某个Servlet的映射路径仅仅为一个正斜杠（即/），那么这个Servlet就将成为当前Web应用程序的缺省Servlet。
